## 1 项目简介

CookHero 是一个面向大众的智能饮食助手系统，提供从厨房准备、菜品学习、营养规划、饮食推荐到个性化计划制定的全流程支持。系统以 Python 为主要开发语言，使用 LangChain 构建检索增强生成、智能代理与向量检索能力，采用 Milvus 作为向量数据库，并提供统一的 REST API 以供前端使用。前端计划使用 React 与 TypeScript 实现用户界面。

系统整合内置菜谱、用户上传菜谱、食堂菜单以及可能的外卖平台数据，通过大语言模型和检索系统为用户提供烹饪知识问答、美食推荐以及饮食计划制定。系统既支持按需查询，也能基于用户长期习惯与健康目标生成深度个性化推荐。

## 2 项目技术栈

本项目的主要技术选型包括以下几个部分：

Python 作为核心开发语言，负责构建服务端逻辑、RAG 管道、数据处理和模型调用。LangChain 用于搭建检索链、数据处理链、智能代理流程与工具调用接口。Milvus 用作向量存储，支持高性能向量检索，适合菜谱文本、图片、多模态 embedding 的管理。后端 Web 服务会封装核心能力并以 REST API 暴露，前端采用 React 与 TypeScript，提供对话界面、菜谱展示界面、推荐界面和用户管理界面。其余基础组件包括关系型数据库用于存储结构化信息，高性能缓存用于提升推荐效率，多模态 embedding 模型负责菜谱图文向量化，语言模型用于回答生成、计划生成与智能路由。

## 3 用户角色与使用场景

CookHero 的用户类型较为丰富，系统设计以满足多类人群的饮食学习、健康管理和高效决策需求。主要用户角色包括厨房小白、健身或减脂用户、关注健康饮食的用户、存在过敏或忌口需求的用户、依赖食堂或外部餐饮的用户以及喜欢自制菜品并希望记录管理的人群。

典型使用场景包括日常菜谱查询，当用户需要低脂高蛋白晚餐、需要了解厨房准备方法、希望学习某道菜的做法时，系统会结合知识库和大模型生成解释与指导。用户也可以通过系统快速获得可选菜品推荐，例如基于当前食堂菜单、忌口条件和饮食偏好筛选出适合的菜品。对于有健康目标的用户，系统能根据基础信息、运动记录和长期目标生成每日或每周饮食计划，并在用户更新生活状态时进行动态调整。系统还能根据菜谱自动生成购物清单，指导用户备菜或储存食材。随着用户的使用和数据积累，系统会逐渐学习用户的口味和行为模式，从而提供更加精准的个性化推荐。

## 4 系统总体架构

CookHero 的总体架构由前端界面层、后端服务层、智能引擎层和数据存储层构成。各层之间通过清晰的接口协作，使系统在保持灵活性的同时具备良好的扩展能力。整体架构围绕 RAG 管道、智能代理、推荐系统和多源数据管理展开，可在后续版本中方便地加入新的模型能力和数据源。

示例：

1. 用户通过对话界面输入自己的需求，比如“我想做一道低脂高蛋白的晚餐，有鸡胸肉和西兰花”，比如"帮我推荐几道适合夏天吃的清淡菜品"，比如“我对花生过敏，能不能帮我推荐一些不含花生的菜谱”，比如"我想了解一下如何准备厨房用具和食材"等等。

2. Query Translation模块可以将用户的查询利用规则匹配或者LLM转换成更适合检索的格式，也可以分成多个子查询。

3. Query Routing模块根据用户的查询意图，决定是否需要检索知识库，或者直接调用LLM生成回答。如果需要检索知识库，则将查询路由到对应的检索模块。（这里路由可以基于规则，也可以基于LLM）

4. Query Construction模块负责将之前的用户查询转换为可检索的格式，比如如果路由到Relational Database检索模块，则需要将查询转换为SQL语句；如果路由到向量数据库检索模块，则需要将查询转换为向量检索的格式（后期可能还会增加图数据库检索模块）。

5. Indexing模块负责将菜谱数据和其他数据进行预处理和索引构建，以便后续的检索使用。这里主要涉及：

   - Chunk Optimization：将长文本进行分块处理，确保每个块的长度适合向量化和检索。

   - Multimodal Embedding： 如果有图片等多模态数据，可以使用多模态嵌入技术将其转换为向量表示。

   - Hybird Embedding：结合稀疏向量和密集向量的优势，提升检索效果。

   - 上下文扩展：利用句子窗口技术，检索使用小块，随后扩展查询和文档的上下文信息，提升检索的相关性。

   - 结构化索引：附加结构化的元数据信息（如菜谱的营养成分、烹饪时间等），在经过滤的、范围更小的文本块集合中，执行向量相似度搜索。

   - 分层索引：创建多个索引，从上层次上组织数据，先进行粗粒度检索（相当于摘要），再进行细粒度检索（相当于主题内容），提高效率和准确性。

6. Re-Ranking和Refinement模块：在初步检索结果的基础上，利用更复杂的模型（如基于Transformer的模型）对结果进行重新排序和精炼，提高最终结果的质量。主要可能涉及：

   - Reciprocal Rank Fusion (RRF)：结合多个检索器的结果，通过计算每个文档的排名得分，进行融合排序。

   - RankLLM / LLM-based Reranker：利用大语言模型对初步检索结果进行重新排序，提升相关性。

   - Cross-Encoder Re-Ranking：使用Cross-Encoder模型对查询和文档对进行评分，提升排序效果。

   - ColBERT Re-Ranking：利用ColBERT模型对初步检索结果进行重新排序，提升相关性。

   - 等等

7. 可以有选择地利用Correction RAG的思路，使用检索评估器 (Retrieval Evaluator)判断每个文档与查询的相关性，并给出“正确 (Correct)”、“不正确 (Incorrect)”或“模糊 (Ambiguous)”的标签。对于正确的，进入知识精炼（Knowledge Refinement）模块，结合多个相关文档生成最终答案；对于不正确的，此时会触发“知识搜索 (Knowledge Searching)”。它会先对原始查询进行“查询重写 (Query Rewriting)”，生成一个更适合搜索引擎的查询，然后进行 Web 搜索，用外部信息来回答问题；对于模糊的，同样会触发“知识搜索”，但通常会直接使用原始查询进行 Web 搜索，以获取额外信息来辅助生成答案。
8. 将最终检索到的信息通过LLM进行整合和生成，形成对用户查询的回答，并返回给用户。

系统的主要数据流从用户输入开始，首先经过查询理解模块，由意图识别和查询转换模型进行分类处理。经过解析的查询会被路由到合适的检索方式，包括向量检索、结构化查询、规则查询或外部数据查询。随后，系统会将检索到的内容与语言模型结合，在生成层完成信息融合与最终回答。对于推荐类请求，系统会在用户画像和菜品元数据的基础上进行策略生成，并返回适合用户的菜品或饮食计划。

系统的主要子模块包括前端交互模块、后端 API 服务模块、查询理解模块、路由模块、RAG 管道、推荐模块、用户画像模块、数据同步模块和索引构建模块。每个模块均保持单一职责，便于维护与替换。RAG 管道负责文本和多模态信息的检索与增强；推荐模块负责计算推荐菜品、食堂选项或每日膳食规划；数据同步模块负责导入用户上传内容与外部来源的菜品信息；索引构建模块负责构建向量索引、元数据结构和混合检索结构。

技术选型上，后端选择以 Python 为主，使用 FastAPI 构建 REST 接口，便于与 LangChain 生态集成；RAG 管道采用 LangChain 和 Milvus 构建混合检索能力，并可自由扩展到稀疏检索、多模态向量或分层索引；智能代理依托 LangChain Agents，用于解析查询、调用工具和分配任务；推荐系统结合规则、统计特征和模型推理，可逐步过渡到个性化推荐模型；关系型数据库用于用户数据与菜品结构化信息，而向量数据库负责存储语义内容与检索向量。

各主要模块承担的职责如下。前端负责展示页面和输入传递；后端负责数据调度与接口封装；查询理解模块负责分析用户输入的需求类型；路由模块负责选择最合适的检索或生成方式；RAG 管道负责知识检索和答案增强；推荐模块负责根据偏好和场景返回适合用户的菜品；用户画像模块整合历史行为与健康信息；数据存储层负责持久化文本、结构化信息和向量表示。整体系统由各模块合作完成，从输入到答案形成清晰的数据流和任务链路。

## 5 模块设计

CookHero 的模块设计遵循“高内聚、低耦合”的架构原则，以清晰的功能分区实现可维护、可扩展的服务体系。各模块之间通过标准化的接口通信，既能独立演化，又能共同支撑系统整体的数据流与业务流程。本节将围绕核心能力介绍模块的设计思路与主要职责。

### 5.1 前端交互模块

前端基于 React 与 TypeScript 实现，由对话界面、菜谱浏览界面、推荐与计划界面、用户管理界面组成。前端与后端通过 REST API 进行交互，负责将用户输入结构化并传递给后端，同时对返回结果进行可视化展示。界面在交互设计上强调简单直观，通过组件化设计便于后期扩展，例如添加图文菜谱预览、交互式营养图表或个性化推荐入口等。

### 5.2 API 服务模块

后端核心以 FastAPI 实现，统一封装查询接口、检索接口、推荐接口、用户管理接口与数据同步接口。API 层负责请求校验、权限控制、日志记录、异常处理以及对下游智能引擎的统一调度。在高并发环境下，可结合异步处理与缓存机制提升性能。所有与用户交互的能力都通过此模块对外暴露，是整个系统的服务入口。

### 5.3 查询理解模块

查询理解模块负责分析用户输入的文本意图，提取任务类型和相关参数。模块内部由意图分类器、参数抽取器与查询标准化组件组成。意图分类器判断用户请求是菜谱问答、步骤解析、营养查询、菜品推荐还是饮食计划制定。参数抽取器从输入中识别食材名称、菜品名称、营养目标、时间范围或用户限制条件等。最终，查询被转换为统一内部格式，便于后续处理。

### 5.4 路由模块

路由模块根据查询类型选择最合适的处理方式，是整个系统的调度器。例如，涉及菜谱内容的查询会进入 RAG 管道，涉及用户状态或健康目标的查询进入推荐模块，涉及任务分解或跨模块操作的查询交由智能代理处理。路由规则既可以基于规则设计，也可以结合轻量级模型实现动态路由，以适应复杂场景。

### 5.5 RAG 管道

RAG 管道是 CookHero 的知识检索与回答生成核心。管道内部包括文本向量检索、多模态检索、检索增强生成、模板化回答生成与检索缓存机制。所有菜谱文本、步骤说明、食材信息和用户上传文档都会存储为向量，通过 Milvus 检索相关内容，再与语言模型结合进行回答融合。对于菜谱图像、食材照片或多模态笔记，系统使用图文模型生成跨模态 embedding，实现更丰富的检索能力。

### 5.6 智能代理模块

智能代理用于处理多步骤、跨模块的复杂任务。例如“帮我制定一周高蛋白低脂计划并生成对应的购物清单”，代理会自动调用查询理解模块、推荐模块和数据处理工具，再整合结果返回给用户。代理模块基于 LangChain Agents 构建，支持工具调用、任务分解、反思与结果验证，使系统具备较高的灵活性。

### 5.7 推荐模块

推荐模块结合规则、统计特征与模型推理，为用户生成个性化菜品推荐。推荐逻辑涵盖食堂菜单筛选、菜品相似度计算、基于偏好的协同过滤、营养目标匹配与历史行为模式分析。对于减脂用户、增肌用户或有特殊饮食偏好的用户，系统会根据用户画像动态调整推荐策略。模块可逐步扩展到深度学习推荐模型或序列建模方法，以提升长期个性化体验。

### 5.8 用户画像模块

用户画像模块整合用户的历史查询记录、口味偏好、营养需求、健康目标、忌口信息与饮食行为模式，构建可持续更新的用户特征向量。模块同时支持显式用户设置与隐式用户行为推理，使系统能够在长期交互中逐渐学习用户的真实偏好，是精准推荐与个性化计划制定的基础。

### 5.9 数据同步模块

数据同步模块负责处理用户上传菜谱、外部菜谱数据、食堂菜单与外卖平台数据。模块会进行格式统一、文本解析、图片处理、元数据抽取与索引更新，确保多源数据进入系统后能够被 RAG 管道和推荐模块直接使用。针对动态数据来源，如食堂每日菜单，模块提供定时同步机制。

### 5.10 索引构建模块

索引构建模块负责生成和更新向量索引、稀疏索引与结构化索引。模块支持增量更新与批量构建，能够在不影响线上服务的情况下完成索引升级。针对多模态内容，会维护独立的多模态 embedding 索引结构；针对复杂查询，可使用混合检索方式提升召回准确率。

## 6 RAG 管道设计与实现

RAG（Retrieval-Augmented Generation）是 CookHero 的核心能力之一，为菜谱问答、烹饪步骤解析、营养解释、菜品对比等任务提供可靠的信息基础。为了适配菜谱类内容的多源、多模态特性，CookHero 的 RAG 管道采用了分层、可扩展的结构设计，支持文本、图像、结构化元数据和外部信息来源的联合检索。

### 6.1 总体流程

RAG 管道的整体数据流从解析后的查询开始，经过检索、候选融合和生成三个阶段。首先，根据查询内容选择合适的检索模式，包括向量检索、关键词检索或混合检索。随后，将检索到的候选内容进行语义过滤和重排序，选择最相关的若干条信息作为上下文，并将上下文与原始查询一同发送给语言模型。最终，模型结合检索信息生成结构化或自然语言回答。

### 6.2 数据准备与向量构建

RAG 的基础是高质量的向量化数据。CookHero 针对不同数据类型构建统一的内容管线，将菜谱步骤、食材说明、烹饪技巧、营养信息及用户上传内容进行预处理和分片。文本会执行分句、去噪和补全元数据等操作，以保证向量的可检索性。对于菜谱图片或用户上传照片，会使用多模态 embedding 模型生成图像向量，使系统能够处理图文混检任务。所有向量及其元数据被同步写入 Milvus，以便后续高效检索。

### 6.3 检索策略设计

CookHero 的检索策略根据不同场景采用不同模式。对于结构化信息，如卡路里、宏量营养素或烹饪时间，会优先使用关键词或结构化查询方式。对于烹饪步骤描述、菜品评价和技巧性知识，会使用向量检索获取语义相关内容。对于复杂查询，例如“低脂高蛋白且不辣的炒菜”，系统会采用混合检索方式，将关键词过滤与向量语义召回结合，提高召回结果的准确性。

### 6.4 结果过滤与重排序

初步检索结果可能包含重复、噪声或片段式内容，因此需要二次过滤与重排序。过滤模块根据查询主题、菜品类别、食材一致性等规则对候选内容进行筛选。重排序模块利用语义匹配模型对候选进行评分，保证最终进入生成阶段的上下文与用户需求高度相关。在需要时，模块还会参考上下文多样性，以避免生成内容过于单一或遗漏关键信息。

### 6.5 上下文组装与回答生成

在候选内容确定后，系统会将其组织为统一格式的上下文，并将上下文与原始查询一并传给语言模型。上下文组织方式依照任务类型调整，例如问答类型采用简洁段落形式，步骤解析采用按顺序列出的步骤结构，营养信息则采用参数化结构。模型结合上下文生成回答结果，并在必要时加入推理解释或提示信息，以提高用户体验。

### 6.6 多模态 RAG

CookHero 对图文类任务提供多模态 RAG 支持。例如用户上传食材照片并询问“我能做什么菜”，系统会将图片编码为向量，并检索与之相关的菜品或食材匹配信息。多模态 RAG 还用于处理带图菜谱、烹饪演示图和图文混合记录，使系统能够在更自然的交流方式中理解用户输入。

### 6.7 缓存与性能优化

为减少重复检索和生成的成本，RAG 管道设有查询与结果缓存策略，对常见菜谱、烹饪方法、热门技巧等内容进行预缓存。在高并发场景下，检索结果缓存能够显著降低频繁调用向量数据库的压力。生成层也会在模型输出稳定的情况下进行缓存，以加快响应速度。

### 6.8 可靠性与可持续迭代

为了提升鲁棒性，RAG 管道设计了降级机制。当向量检索失败时，会自动回退到关键词检索；当生成模型负载过高时，会暂停复杂查询并优先保证问答类请求。整个 RAG 结构为后续扩展预留接口，能够方便地加入新的 embedding 模型、rerank 模型或结构化知识源，实现持续迭代。

## 7 推荐系统设计与实现

推荐系统是 CookHero 提供个性化菜品和饮食计划的核心模块。它结合用户画像、历史行为、菜品特征与健康目标，为不同用户提供精确、高效的推荐服务。

### 7.1 系统目标

推荐系统的目标包括：

- 提供个性化菜品推荐，满足用户口味、营养需求和饮食限制。
- 支持基于食堂菜单、用户上传菜品和外部平台数据的多源推荐。
- 动态调整每日或每周饮食计划，以配合用户的运动、健康目标和行为变化。
- 提供可解释的推荐理由，提高用户信任和系统可用性。

### 7.2 数据来源与处理

推荐系统依赖多源数据，包括：

- **内置菜谱与用户上传菜谱**：用于建立菜品特征向量和类别标签。
- **食堂与外卖平台数据**：通过爬虫或 API 获取实时菜单信息。
- **用户画像与行为数据**：记录历史查询、选择偏好、饮食限制和健康目标。

数据经过标准化、清洗、特征提取和 embedding 转换后进入推荐模块。文本和图像内容分别通过向量化模型生成 embedding，用于相似度计算与召回。

### 7.3 推荐策略

推荐策略采用混合方法：

- **规则与约束策略**：基于用户忌口、过敏信息和营养需求进行硬过滤。
- **基于特征的相似度推荐**：通过菜品特征 embedding 计算相似度，为用户推荐相似菜品。
- **协同过滤与行为预测**：根据用户历史行为和相似用户偏好进行推荐。
- **健康目标匹配**：结合用户目标（减脂、增肌、均衡饮食）和运动数据生成每日或每周饮食计划。
- **动态调整**：系统根据实时用户反馈或新的行为记录更新推荐结果。

### 7.4 输出与交互

推荐结果包括菜品列表、个性化饮食计划、营养数据和购物清单。系统可在前端界面显示推荐理由和匹配指标，增强用户信任。推荐模块还与 RAG 管道协作，提供解释性说明或烹饪指导。

### 7.5 性能与扩展

为应对高并发和大规模用户，推荐模块设计了缓存策略和批量计算能力。系统接口支持扩展到深度学习模型、序列推荐模型或强化学习策略，以实现长期个性化优化。

## 8 智能代理系统设计

智能代理系统是 CookHero 中处理复杂、多步骤任务的核心能力模块，负责跨模块调度和任务分解，使系统能够完成多目标、跨模块的智能操作。

### 8.1 系统目标

智能代理系统的设计目标包括：

- 自动处理复杂任务，如“生成一周饮食计划并生成购物清单”。
- 调度和组合多个模块能力，包括 RAG 管道、推荐模块和数据处理工具。
- 支持任务分解、反思和自我纠错，提高结果可靠性。
- 提供灵活扩展接口，可集成新的工具或外部服务。

### 8.2 核心能力

智能代理具备以下核心能力：

- **任务解析与分解**：将用户的复合查询拆解为多个子任务，并分配给对应模块处理。
- **工具调用**：根据任务需求调用 RAG 管道、推荐模块或外部接口。
- **结果整合**：将不同子模块返回的结果进行融合、推理或格式化，生成完整响应。
- **自我纠错与优化**：在子任务失败或结果不理想时，自动触发重试、替代方案或查询重写。

### 8.3 实现方式

智能代理基于 LangChain Agents 构建，利用以下特性：

- **多工具支持**：可以灵活调用检索、推荐、计算、计划生成等模块。
- **动态任务调度**：根据任务复杂度和实时状态决定执行顺序。
- **策略扩展**：可逐步增加新的任务类型、复杂度或交互逻辑。
- **可解释性**：记录代理决策过程，便于后续分析和优化。

### 8.4 应用示例

智能代理可处理的任务示例包括：

- 根据用户目标生成个性化一周饮食计划，并输出每日餐点和营养信息。
- 为用户推荐食堂可选菜品，同时生成营养分析和过敏提醒。
- 整合多来源信息，回答复杂问题，例如“我想在夏天做清淡高蛋白菜品，并且避免花生”。

通过智能代理，CookHero 能够实现从单一查询到跨模块综合任务的自动处理，提升用户体验和系统智能水平。

## 9 非功能性需求与系统性能设计

非功能性需求确保 CookHero 系统在功能之外具备可靠性、效率和可持续发展能力，是整个系统设计的重要组成部分。

### 9.1 性能与响应能力

CookHero 需要在高并发环境下保证良好的响应速度，具体要求包括：

- **查询响应时间**：普通菜谱问答或推荐请求在 1~2 秒内返回结果，高复杂度生成任务在 3~5 秒内完成。
- **吞吐量**：系统需支持至少数百到上千并发用户访问。
- **检索效率**：向量检索模块（Milvus）应优化索引结构和检索策略，确保高召回率同时降低延迟。
- **缓存机制**：对热门查询、常用菜谱和生成内容使用缓存，加快响应速度，减少重复计算。

### 9.2 可扩展性

系统设计需支持功能和用户规模的扩展：

- **模块化架构**：各模块独立部署，可单独升级或扩展，例如 RAG 管道、推荐模块或智能代理。
- **多源数据扩展**：支持导入新的菜谱数据源、食堂 API 或外卖平台数据，无需修改核心逻辑。
- **模型升级**：语言模型、向量化模型和推荐模型可平滑替换，支持后续性能提升或新算法接入。

### 9.3 可用性与可靠性

为了保证系统在不同环境下稳定运行：

- **故障隔离**：模块出现异常时，系统可降级处理或提供部分服务，避免整体中断。
- **容错机制**：RAG 管道和智能代理具备降级策略，例如向量检索失败时回退关键词检索。
- **健康监控**：后端服务提供日志和状态监控，便于运维人员及时发现问题。

### 9.4 安全性与隐私保护

CookHero 涉及用户健康信息和饮食偏好，需要严格保护用户数据：

- **数据传输安全**：API 使用 HTTPS 加密通信，防止数据泄露。
- **用户数据隔离**：不同用户的历史记录和偏好数据独立存储，确保隐私。
- **访问控制**：对 API 接口进行权限管理，限制敏感操作和数据访问。
- **敏感信息处理**：用户上传的照片、个人健康数据等在存储和分析过程中进行脱敏处理。

### 9.5 可维护性与可测试性

系统设计需支持长期维护和快速迭代：

- **模块独立测试**：各模块提供单元测试和集成测试接口，确保升级不破坏现有功能。
- **日志与追踪**：关键流程（如 RAG 查询、推荐生成、智能代理任务）记录详细日志，便于排错和性能分析。
- **配置化管理**：系统参数、模型路径和推荐策略均可通过配置文件或管理接口调整，无需修改代码。

### 9.6 用户体验与可用性

非功能性需求同样关注用户体验：

- **界面响应流畅**：前端交互界面操作顺滑，输入反馈及时。
- **结果可解释性**：推荐结果和生成回答提供解释或参考来源，提高信任感。
- **多终端适配**：系统可在桌面端、移动端或平板设备上良好运行。

通过上述非功能性设计，CookHero 能够在保证功能丰富的前提下，提供稳定、安全、高效且可持续迭代的服务，为用户提供长期可靠的饮食助手体验。
