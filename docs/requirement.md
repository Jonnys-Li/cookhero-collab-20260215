# CookHero: 需求规格与项目规划

---

## 第一部分：需求规格说明书 (Requirements Specification Document)

### 1. 项目概述

CookHero 是一个面向大众的智能饮食助手系统，提供从厨房准备、菜品学习、营养规划到个性化推荐的全流程支持。系统以 Python 为主要开发语言，使用 LangChain 构建检索增强生成（RAG）能力，采用 Milvus 作为向量数据库，并通过统一的 REST API 对外提供服务。

**核心价值**:

*   **智能化**: 利用 LLM 理解复杂的自然语言查询，并生成高质量的文本。
*   **知识驱动**: 基于 RAG 架构，确保所有回答都源自于可靠、专业的知识库，而非模型凭空想象。
*   **高相关性**: 通过多阶段的检索、过滤和重排（Reranking）机制，最大化上下文与用户查询的相关性。
*   **可扩展性**: 采用模块化和插件式设计，便于未来集成更多知识源和AI模型。

### 2. 功能性需求 (Functional Requirements)

#### 2.1. 数据处理与入库 (Data Ingestion) ✅

*   **FR-1**: ✅ 支持多种异构数据源，已实现菜谱 (`dishes`)、烹饪技巧 (`tips`) 和通用文本 (`generic_text`)。
*   **FR-2**: ✅ 数据源采用可插拔的 `BaseDataSource` 接口设计。
*   **FR-3**: ✅ 支持为每个数据源创建独立的 Milvus 集合（Collection）。
*   **FR-4**: ✅ 支持为文档生成唯一的、基于文件路径的确定性 ID。
*   **FR-5**: ✅ 支持混合索引，同时创建稠密向量和稀疏向量 (BM25)。
*   **FR-6**: ✅ 实现菜谱索引文档功能，包含所有菜谱名称，用于推荐类查询。

#### 2.2. RAG 核心服务 (`RAGService`) ✅

*   **FR-6**: ✅ 实现查询重写（Query Rewriting），将模糊查询转化为更精确的搜索指令。
*   **FR-7**: ✅ 支持并行检索（Parallel Retrieval），从所有数据源同时检索文档。
*   **FR-8**: ✅ 支持混合搜索（Hybrid Search），结合语义与关键词进行检索，并能根据查询意图动态调整权重。
*   **FR-9**: ✅ 支持"Small-to-Large"检索模式，检索小块文本，返回完整文档。
*   **FR-10**: ✅ 支持可插拔的 `BaseReranker` 重排序模块。
*   **FR-11**: ✅ 已实现基于 `SiliconFlow` 专用 API 的高效重排序器。
*   **FR-12**: ✅ 实现检索结果排序和过滤，在 rerank 前按 score 排序并截取 top_k。
*   **FR-13**: ✅ 支持智能排序器选择，根据查询特征动态调整混合搜索权重。
*   **FR-14**: ✅ 实现推荐类查询检测，自动增加检索数量以获取更多样化的结果。
*   **FR-15**: ✅ 优化查询重写 prompt，扩展推荐类查询概念（如"荤素搭配"）。

#### 2.3. API 接口 (`chat` endpoint) ✅

*   **FR-14**: ✅ 提供统一的 `/api/v1/chat` API 端点用于接收自然语言查询。
*   **FR-15**: ✅ API 支持流式（Streaming）和非流式响应。

#### 2.4. 智能代理与高级 RAG (未来规划) 🚧

*   **FR-16**: 🚧 [规划中] 实现一个智能代理（Agent），用于处理多步骤的复杂任务（如"制定一周健身计划并生成购物清单"）。
*   **FR-17**: 🚧 [规划中] 引入 Correction RAG 流程，包括：
    *   一个用于判断检索内容相关性的"检索评估器"。
    *   一个在检索内容不佳时，触发查询重写和 Web 搜索的"知识搜索"模块。
*   **FR-18**: 🚧 [规划中] 支持多模态 RAG，能够处理用户上传的图片查询。
*   **FR-19**: 🚧 [规划中] 引入结构化数据查询能力，能将自然语言转换为 SQL 等查询语言。

---

### 3. 非功能性需求 (Non-Functional Requirements)

*   **NFR-1 (性能)**: ✅ 响应延迟应在可接受范围内，流式请求能快速返回首个 token。已实现并行检索和结果预过滤优化。
*   **NFR-2 (可扩展性)**: ✅ 系统通过 `BaseDataSource` 和 `BaseReranker` 等模块的抽象基类设计，支持未来轻松扩展新的数据源和 AI 模型。
*   **NFR-3 (可维护性)**: ✅ 所有关键模块都有清晰的日志记录，并提供容器化部署方案（Docker Compose）。
*   **NFR-4 (可测试性)**: ✅ 提供 `tests/test_rag.py` 脚本，用于验证 RAG 管道核心功能的正确性。

## 第二部分：项目计划 (Development Process & Plan)

### 里程碑 1: 核心 RAG 管道搭建 (MVP) - ✅ 已完成

*   **目标**: 验证核心 RAG 能力，实现基于单一数据源（菜谱）的问答。
*   **核心交付物**:
    *   ✅ **`HowToCookDataSource`**: 实现了菜谱数据的加载和处理。
    *   ✅ **Milvus 集成**: 能够将文档向量化并存入 Milvus。
    *   ✅ **基础 RAG 服务**: `RAGService` 实现基本的"检索-生成"流程。
    *   ✅ **FastAPI 接口**: 提供了支持流式响应的 `/chat` 端点。
    *   ✅ **确定性 ID**: 解决了因随机 ID 导致的数据映射失败问题。

### 里程碑 2: 多源支持与并行检索 - ✅ 已完成

*   **目标**: 扩展 RAG 管道，使其能够处理多种类型的知识，并支持并行检索。
*   **核心交付物**:
    *   ✅ **`BaseDataSource`**: 抽象出数据源基类，提升了可扩展性。
    *   ✅ **`TipsDataSource`**: 新增了烹饪技巧数据源。
    *   ✅ **`GenericTextDataSource`**: 新增了通用文本数据源，支持句子窗口索引。
    *   ✅ **多集合支持**: 数据入库脚本和 RAG 服务被重构，能够为不同数据源使用不同的 Milvus 集合。
    *   ✅ **并行检索**: 实现从所有数据源并行检索文档，提升响应速度。

### 里程碑 3: 上下文质量优化 (Reranker) - ✅ 已完成

*   **目标**: 通过引入重排序步骤，进一步提升提供给 LLM 的上下文质量，减少无关信息的干扰。
*   **核心交付物**:
    *   ✅ **`BaseReranker`**: 抽象出重排序器基类。
    *   ✅ **`SiliconFlowReranker`**: 实现了基于专用 rerank API 的高效重排序器。
    *   ✅ **可配置化**: 在 `config.yml` 中增加了 `reranker` 配置节，可轻松启用/禁用该功能。
    *   ✅ **`RAGService` 集成**: 将 reranker 步骤无缝集成到 `ask` 方法的检索流程中。

### 里程碑 4: 性能优化与智能检索 - ✅ 已完成

*   **目标**: 优化检索性能，减少 rerank 计算量，提升系统响应速度。
*   **核心交付物**:
    *   ✅ **智能排序器选择**: 根据查询特征（关键词 vs 语义）动态调整混合搜索权重。
    *   ✅ **检索结果预过滤**: 在 rerank 前按检索 score 排序并截取 top_k，减少 rerank 输入量。
    *   ✅ **分数传递机制**: 确保检索分数正确传递到父文档，支持精确排序。
    *   ✅ **去重优化**: 在去重时保留分数更高的文档。

### 里程碑 5: 推荐查询优化与菜谱索引 - ✅ 已完成

*   **目标**: 优化推荐类查询的检索效果，提供更准确的菜品推荐。
*   **核心交付物**:
    *   ✅ **菜谱索引文档**: 创建包含所有菜谱名称的索引文档，按类别组织。
    *   ✅ **索引 Chunk 优化**: 索引 chunk 仅包含推荐相关关键词，提高语义匹配准确性。
    *   ✅ **推荐查询检测**: 自动识别推荐类查询，增加检索数量（2倍）以获取更多样化结果。
    *   ✅ **查询重写增强**: 改进查询重写 prompt，扩展推荐类查询概念（如"荤素搭配"→"既有肉类又有蔬菜"）。
    *   ✅ **智能排序器扩展**: 扩展语义查询指示器，正确识别推荐类查询。

### 里程碑 6: 高级 RAG 与智能化 (🚧 规划中)

*   **目标**: 引入更强的鲁棒性和自主性，使系统能处理更复杂的任务并从外部获取知识。
*   **核心规划**:
    *   🚧 **Correction RAG**: 实现"检索评估器"和"知识搜索"（Web Search）功能，当内部知识不足时，能主动进行网络搜索以补全信息。
    *   🚧 **智能代理 (Agent)**: 基于 `LangChain Agents` 构建一个能够分解任务、调用工具（如 RAG、推荐模块）的智能代理，以完成多步骤的复杂指令。
    *   🚧 **用户画像模块**: 构建持久化的用户画像系统，用于实现深度个性化推荐和计划制定。
    *   🚧 **多模态能力**: 扩展 RAG 管道以支持图像输入，实现"以图搜菜"等功能。

### 里程碑 7: 推荐系统 (🚧 规划中)

*   **目标**: 提供个性化菜品和饮食计划推荐。
*   **核心规划**:
    *   🚧 **用户偏好学习**: 基于历史查询和交互行为学习用户口味。
    *   🚧 **营养目标匹配**: 根据用户健康目标（减脂、增肌等）推荐菜品。
    *   🚧 **饮食计划生成**: 生成每日或每周饮食计划。
    *   🚧 **购物清单生成**: 根据推荐菜品自动生成购物清单。

### 里程碑 8: 前端界面 (🚧 规划中)

*   **目标**: 提供用户友好的 Web 界面。
*   **核心规划**:
    *   🚧 **对话界面**: React + TypeScript 实现的聊天界面。
    *   🚧 **菜谱浏览**: 可视化展示菜谱和烹饪步骤。
    *   🚧 **推荐界面**: 展示个性化推荐和饮食计划。
    *   🚧 **用户管理**: 用户注册、登录、偏好设置。

## 第四部分：技术债务与未来改进

### 技术债务

1. **缓存机制**: 目前未实现查询和结果缓存，未来需要添加 Redis 缓存层。
2. **错误处理**: 需要更完善的错误处理和降级机制。
3. **监控与日志**: 需要集成 APM 工具（如 Prometheus、Grafana）进行性能监控。
4. **单元测试**: 需要增加更多单元测试覆盖核心模块。

### 未来改进方向

1. **性能优化**:
   - 实现查询结果缓存
   - 优化向量检索性能
   - 支持批量查询

2. **功能增强**:
   - 多模态 RAG（图像输入）
   - 结构化数据查询（SQL生成）
   - 实时数据同步

3. **用户体验**:
   - 前端界面开发
   - 个性化推荐
   - 用户画像系统

4. **系统可靠性**:
   - 分布式部署支持
   - 高可用架构
   - 数据备份与恢复
