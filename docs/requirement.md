# CookHero 项目需求文档

本文档详细列出了 CookHero 项目需要开发的功能需求、非功能需求，并提供了一个整体的开发流程与计划。

## 1. 功能需求 (Functional Requirements)

### 1.1 前端交互模块

- **对话界面**:
  - 提供一个用户友好的聊天界面，用于输入文本查询。
  - 能够展示系统返回的文本、图片和结构化数据（如菜谱卡片、营养列表）。
  - 支持多轮对话，维持上下文。
- **菜谱浏览与搜索**:
  - 提供专门的页面用于浏览和搜索内置菜谱及用户上传的菜谱。
  - 搜索功能支持关键词、食材、菜系等多种筛选条件。
- **推荐与计划展示**:
  - 在指定区域展示个性化推荐的菜品或饮食计划。
  - 饮食计划应以日历或列表形式清晰展示每日的餐点安排。
- **用户管理界面**:
  - 用户可以注册和登录。
  - 用户可以设置个人信息，如健康目标（减脂、增肌）、饮食偏好、过敏信息、忌口等。
- **数据上传与管理**:
  - 用户可以上传自己的菜谱（支持文本和图片）。
  - 用户可以查看和管理自己上传过的内容。

### 1.2 后端 API 服务模块 (FastAPI)

- 提供统一的 RESTful API 接口，覆盖所有系统功能。
- **用户认证与管理**:
  - /register: 用户注册。
  - /login: 用户登录。
  - /profile: 获取和更新用户个人信息。
- **核心交互接口**:
  - /chat: 处理用户的对话输入，并根据意图分发到不同模块。
- **RAG 与知识问答**:
  - /rag/query: 接收查询，通过 RAG 管道返回知识性回答。
- **推荐系统接口**:
  - /recommend/dishes: 根据用户画像和上下文推荐菜品。
  - /recommend/plan: 生成每日或每周饮食计划。
- **数据管理接口**:
  - /recipes: 获取菜谱列表，支持分页和过滤。
  - /recipes/{id}: 获取单个菜谱详情。
  - /upload/recipe: 用户上传新菜谱。

### 1.3 RAG 管道

- **多源数据接入**:
  - 支持内置菜谱、用户上传菜谱、食堂菜单、外部平台数据。
- **数据预处理与索引**:
  - 文本分块 (Chunking) 与优化。
  - 结构化元数据提取（如营养成分、烹饪时间）。
  - 使用多模态模型为图文内容生成向量 (Embedding)。
  - 构建并维护 Milvus 中的向量索引和元数据索引。
- **混合检索能力**:
  - 支持向量语义检索。
  - 支持基于元数据的关键词过滤检索。
  - 融合多种检索结果以提高召回率和准确率。
- **结果重排与精炼 (Re-Ranking & Refinement)**:
  - 对初步检索结果进行二次排序，提升与查询的相关性。
- **上下文生成与答案合成**:
  - 将最相关的检索结果组装成上下文。
  - 调用大语言模型 (LLM) 结合上下文生成流畅、准确的回答。

### 1.4 推荐系统模块

- **多种推荐策略**:
  - **基于规则**: 根据用户的过敏信息、忌口等进行硬性过滤。
  - **基于内容**: 根据菜品的特征（如食材、口味、菜系）推荐相似菜品。
  - **协同过滤**: 根据用户历史行为和相似用户的偏好进行推荐。
  - **基于健康目标**: 结合用户的减脂、增肌等目标和营养学知识生成饮食计划。
- **动态与可解释性**:
  - 推荐结果能根据用户的实时反馈动态调整。
  - 提供推荐理由，增强用户信任感。

### 1.5 智能代理 (Intelligent Agent)

- **复杂任务处理**:
  - 能够理解并执行多步骤的复杂指令，如“帮我制定一个为期三天、低热量、高蛋白的午餐计划，并生成购物清单”。
- **工具调用与编排**:
  - 能够自主调用 RAG、推荐系统、数据查询等内部工具。
  - 能够编排工具的调用顺序以完成任务。
- **自我反思与纠错**:
  - 在任务执行失败或结果不理想时，具备一定的反思和重试能力。

### 1.6 用户画像模块

- **数据收集**:
  - 显式收集：用户在个人资料中填写的偏好、目标等。
  - 隐式收集：用户的查询历史、点击行为、收藏的菜谱等。
- **画像构建**:
  - 将收集到的数据整合成结构化的用户标签和特征向量。
  - 画像需要支持持续更新。

## 2. 非功能需求 (Non-Functional Requirements)

- **性能**:
  - 普通查询响应时间应在 2 秒以内。
  - 复杂生成任务（如周计划）响应时间在 5 秒以内。
- **可扩展性**:
  - 系统采用模块化设计，方便独立升级和替换（如更换 LLM、Embedding 模型）。
  - 支持方便地接入新的数据源。
- **可用性与可靠性**:
  - 系统应具备高可用性，关键模块具备故障降级和容错能力。
  - 提供健康监控和日志记录，便于问题排查。
- **安全性与隐私**:
  - 所有数据传输使用 HTTPS 加密。
  - 严格保护用户个人信息和隐私数据，进行必要的脱敏处理。
- **可维护性**:
  - 代码结构清晰，遵循编码规范。
  - 提供完善的单元测试和集成测试。
  - 关键参数可配置。

## 3. 开发流程与计划 (Development Process & Plan)

项目将采用敏捷开发的模式，分阶段进行迭代。**核心原则是优先完成所有后端核心功能的开发与测试，确保逻辑的稳定与健壮后，再进行前端页面的开发与集成。**

### 阶段一：核心后端与 RAG 管道搭建

- **目标**: 搭建项目的基本骨架，实现一个基于 RAG 的菜谱问答 MVP，并通过自动化测试验证其功能。
- **任务**:
  1.  **环境搭建**: 初始化 Python 项目，使用 FastAPI 搭建 Web 服务框架。
  2.  **数据库设置**: 安装并配置 Milvus 和关系型数据库（如 SQLite）。
  3.  **数据模型定义**: 定义菜谱、用户等核心数据结构的 Schema。
  4.  **初始数据导入**: 编写脚本，将初始菜谱数据进行处理、向量化并存入数据库。
  5.  **基础 RAG 管道实现**: 实现 `接收查询 -> 向量检索 -> LLM 生成 -> 返回答案` 的核心逻辑。
  6.  **基础 API 定义**: 创建内部测试用的 API 端点，此时不追求与前端完全匹配。
  7.  **单元与集成测试**: 为数据导入和 RAG 核心逻辑编写完备的测试用例。

### 阶段二：用户系统与推荐功能开发

- **目标**: 在后端实现完整的用户系统和第一版个性化推荐功能。
- **任务**:
  1.  **用户认证与画像**: 实现用户注册、登录、JWT 认证以及用户画像的存储与更新逻辑。
  2.  **基础推荐系统**: 实现基于规则（忌口过滤）和基于内容（相似菜品）的推荐算法。
  3.  **后端逻辑集成**: 将用户画像与推荐系统关联，使其能根据用户偏好进行推荐。
  4.  **测试**: 通过 API 调用和自动化测试，验证用户系统和推荐功能的正确性。

### 阶段三：智能代理与高级功能开发

- **目标**: 引入智能代理，处理复杂的、多步骤的用户请求。
- **任务**:
  1.  **智能代理设计与实现**: 使用 LangChain Agents 设计并实现代理，定义其可使用的工具集（如 `search_recipe`, `recommend_dishes`）。
  2.  **复杂任务实现**: 实现一个核心的复杂任务，例如“生成一周饮食计划”，并验证其结果的合理性。
  3.  **后端路由逻辑**: 完善后端的总控制逻辑，使其能根据用户输入智能判断调用 RAG、推荐模块还是智能代理。
  4.  **端到端测试**: 对所有后端功能（RAG、推荐、代理）进行端到端的集成测试。

### 阶段四：全功能集成与 API 定型

- **目标**: 完成所有后端功能的深度集成，并根据测试结果最终确定提供给前端的 API 接口规范。
- **任务**:
  1.  **后端功能联调**: 确保 RAG、推荐、用户、代理等所有模块协同工作无误。
  2.  **API 接口最终确定**: 根据联调结果，设计并固化最终的 RESTful API 规范，包括请求/响应格式、错误码等。
  3.  **性能与安全**: 进行初步的性能测试和安全检查，增加缓存等优化措施。
  4.  **API 文档撰写**: 编写详细的 API 文档（如 Swagger/OpenAPI）。

### 阶段五：前端页面实现与最终集成

- **目标**: 开发前端界面，并与已经稳定和文档化的后端 API 对接，完成产品的最终形态。
- **任务**:
  1.  **前端项目初始化与 UI 开发**: 创建 React 项目，开发所有必要的 UI 组件和页面（对话页、用户设置页等）。
  2.  **API 对接**: 严格按照第四阶段定型的 API 文档，进行前后端对接。
  3.  **端到端测试**: 进行完整的前后端联合测试，确保用户体验流畅。
  4.  **UI/UX 优化**: 根据测试反馈，对界面交互和视觉效果进行调整。

### 阶段六：最终优化、文档与部署

- **目标**: 对整个系统进行最终优化，完善所有文档，并准备生产部署。
- **任务**:
  1.  **容器化**: 编写 `Dockerfile` 和 `docker-compose.yml`，实现项目的一键部署。
  2.  **部署演练**: 在测试环境中进行部署演练。
  3.  **最终文档**：完善 `README.md`，提供最终的部署和运行指南。
